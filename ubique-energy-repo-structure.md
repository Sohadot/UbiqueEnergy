## What Each Layer Does

### DOCTRINE/
**Purpose:** Define the belief system
- **energy-topology-doctrine.md** — Core philosophy about energy systems
- **inevitability-framework.md** — How this asset becomes unmistakable
- **strategic-thesis.md** — Why UbiqueEnergy is the right name & direction

### ASSETS/ubiquenergy-com/
**Purpose:** All asset-specific strategy & execution

#### UbiqueEnergy_DOSSIER.md
Master document that governs everything:
- Asset identity & positioning
- Problem space definition
- Conceptual foundation
- Reference architecture blueprint
- Authority building strategy
- Institutional embedding path
- Publication & monetization gates
- Current status & next actions

#### PROBLEM_SPACE.md
Detailed analysis:
- What's broken in energy thinking?
- Why current frameworks fail
- What UbiqueEnergy solves
- Why this domain is the answer

#### CONCEPTUAL_FOUNDATION.md
Core definitions & architecture:
- Energy as Topology (definition)
- Distributed Intelligence (framework)
- Energy-Information Convergence (thesis)
- Complete taxonomy of concepts
- How concepts relate to each other

#### REFERENCE_ARCHITECTURE.md
What we build:
- Homepage structure & content
- Manifesto content & positioning
- Definitions & glossary scope
- Framework pages & interactive tools
- Case studies selection
- Article clusters
- Database specifications

#### INSTITUTIONAL_PATH.md
How it becomes essential:
- Where engineers will discover it
- How researchers will cite it
- Which companies will use it
- What decisions it influences
- SEO strategy (organic discovery)
- Network integration points

#### SITE/ (Public Digital Asset)
The actual built website:
- All HTML/CSS/JS files
- All content pages
- Databases & data files
- Interactive tools
- Visual assets

### SKILLS/
**Purpose:** How-to modules for execution
Each skill explains:
- What to do
- How to do it
- Quality standards
- Examples

### TEMPLATES/
**Purpose:** Standardized document formats
Ensures consistency across:
- Page structures
- Definition writing
- Case studies
- Articles
- Frameworks

### OPERATING-RULES/
**Purpose:** Governance & discipline
Prevents drift with:
- Quality standards
- Publication gates
- Content discipline
- Naming rules
- Update protocols

## Entry Point

Start here:
1. Read: `README.md`
2. Understand: `DOCTRINE/energy-topology-doctrine.md`
3. Study: `ASSETS/ubiquenergy-com/UBIQUENERGY_DOSSIER.md`
4. Build: Follow `REFERENCE_ARCHITECTURE.md`
5. Govern: Apply rules from `OPERATING-RULES/`

---

## Key Principle

**Every file serves a purpose.**

If a file exists, it must:
- Govern something
- Clarify something
- Build something
- Document something

Random files degrade the system.

---

## Update Protocol

Only update when:
1. Strategic decision changes
2. New phase of building starts
3. Major content complete
4. Learnings need capture

Never update for activity. Only update for clarity.

---

## System Integrity

The system works **only if all layers are respected:**

- DOCTRINE defines what we believe
- ASSETS applies doctrine to this specific asset
- SKILLS explains how to execute
- TEMPLATES standardize outputs
- OPERATING-RULES prevent drift

Mix layers → system degrades.
Respect layers → system compounds.
